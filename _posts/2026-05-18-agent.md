The past few weeks have been crazy busy with an ongoing SciML project. I hope to share something interesting soon. In the meantime, I've been setting up "Vibe Physics" (https://lnkd.in/geQgPM7v) for my own use. I've been using VS Code + GitHub Copilot mostly for vibe coding, which has been working really well. Then, as I mentioned in a previous post, I came across Matthew Schwartz's blog post about AI research assistants and decided to give it a try.

The idea is simple. My pipeline has four components: Python code, training config files in YAML format, a data folder containing various CFD simulation results, and a training/inference results folder. I prepared a copilot-instructions.md in each folder. In the code folder, I gave an overview of the codebase (along with a couple of papers in PDF format), described what each script does, and — since my training runs on HPC with LSF — explained my LSF script and its arguments. The config folder's file includes a template YAML with the meaning of each key and a description of naming conventions. The data folder has a markdown describing which subfolder contains which dataset, since I have many different training sets.

Preparing these markdowns took considerable effort. For the code documentation, I asked Claude to analyze my codebase and write it up — that part was fairly straightforward. But I had to provide detailed context on the LSF queue setup and my submission script, which has grown over many years. I took time and care to document my training configurations and data collections as thoroughly as I could, and had Claude review everything afterward for consistency. The most time-consuming part was the inference results folder, where I described (again, with Claude's help) all post-analysis files, mostly in CSV format.

I used to keep a couple of Excel files tracking my various runs — different datasets, hyperparameters, ablation studies, the whole nine yards. Even then, I frequently lost track of why a particular job was launched. Not anymore. Copilot now understands all the runs in my queue, monitors their progress, and generates a performance metric comparison when they complete. It makes recommendations based on its findings, and with a simple "Let's do it," it populates the config YAML files for the next iteration, submits jobs to the queue, and adds entries to the log markdown.

I have a very challenging project at hand, one that seemed impossible to meet on deadline just a few weeks ago. Matthew's "AI grad student" has been a real force multiplier — boosting my productivity many times over. I can imagine the same principle applied to any engineering CAE workflow. The way we do CAE may look very different, soon. A quick snippet of post-analysis markdown is shown.
